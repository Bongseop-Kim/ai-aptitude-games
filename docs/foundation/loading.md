# Loading

일관된 로딩 UI 선택을 위한 가이드.

## 원칙

- 로딩 자체를 줄이는 것이 최우선이다.
- 로딩이 불가피하면 사용자가 진행 중임을 분명히 알 수 있어야 한다.
- 예상 로딩 시간과 화면 구조에 맞는 로딩 요소를 선택한다.

## 로딩 요소

| 요소 | 용도 | 적합 시간 |
| --- | --- | --- |
| Progress Circle | 짧은 비동기 작업, 버튼 내부 제출 상태, 레이아웃 예고가 필요 없는 상태 | 1-10초 |
| Progress Bar | 단계나 진행률이 있는 작업 | 4초 이상 |
| Skeleton | 목록, 카드, 리포트처럼 콘텐츠 구조를 미리 보여줄 수 있는 화면 | 1-10초 |

## 시간별 권장

| 로딩 시간 | 처리 |
| --- | --- |
| 1초 이내 | 표시하지 않는다. 표시하면 깜빡임이 생긴다. |
| 1-4초 | Progress Circle 또는 Skeleton |
| 4-10초 | Skeleton 또는 Progress Bar, 필요하면 안내 문구 |
| 10초 이상 | 진행률, 취소, 재시도, 백그라운드 처리 가능성을 검토 |
| 1분 이상 | 백그라운드 처리와 명확한 상태 회복 경로를 제공 |

## 상황별 선택

| 상황 | 권장 요소 |
| --- | --- |
| 앱 첫 진입 | Progress Circle, Skeleton |
| 화면 전환 | Skeleton, Progress Circle |
| 무한 스크롤/추가 로드 | 작은 Progress Circle |
| Pull to refresh | 플랫폼 기본 refresh indicator |
| 로그인/저장/제출 | 버튼 내부 Progress Circle 또는 disabled 상태 |
| 리포트/분석 데이터 로드 | Skeleton 우선 |

## 실패 처리

로딩이 실패하면 기술 오류를 그대로 노출하지 않는다.

- 무엇이 안 됐는지 사용자 관점으로 설명한다.
- 다시 시도할 수 있는 액션을 제공한다.
- 입력이나 진행 상태가 유지되는지 명확히 한다.

문구 작성은 [writing](./writing.md)의 오류 메시지 규칙을 따른다.
